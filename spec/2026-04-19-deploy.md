## Current Scope

This spec is only for `lexe deploy`.

It assumes the target VM already exists and has been prepared by `lexe provision`. This spec covers
the local-to-remote deploy flow for a provisioned exe.dev VM, including image build/transfer,
container replacement, and the current health-gated startup path.

### Initial Implementation Phase

The first implementation is intentionally narrowed to the minimum needed to deploy
`examples/hello`:

- local image build
- image transfer to the provisioned VM
- remote `docker compose` startup using local `compose.yaml` and `compose.server.yaml`
- container health via Docker Compose `--wait` and a compose-defined healthcheck

Deferred from this first phase:

- secrets hydration
- pre-start and post-start hooks
- local HTTP healthcheck polling beyond Docker Compose health waiting
- deploy log writing
- remote image prune
- already-deployed no-op behavior and `--force`

## Decisions Captured

- `deploy` does not create or repair the VM; the user runs `provision` first.
- `deploy` requires a clean git working tree by default.
- `deploy --allow-dirty` permits a dirty working tree for intentional non-release deploys.
- Release tags use `vYYYY-MM-DD-<sha7>` from the current git commit SHA.
- Dirty-worktree deploy tags append `-dirty` to the computed release tag.
- The app image is built locally and then transferred to the VM.
- Remote container lifecycle is managed from the local machine against the provisioned VM.
- Image transfer targets the VM with `docker pussh <vm-host>`.
- Remote compose targets the VM through Docker's SSH host support rather than Docker contexts.
- The current success criterion is Docker Compose health waiting against the service healthcheck.
- Deploys must always use `compose.yaml` and `compose.server.yaml` explicitly.
- `compose.override.yaml` is never part of a deploy.

## Deploy-Specific Config Surface

This spec builds on the minimum config already established for provision work.

The current implementation does not add deploy-specific config beyond the provision phase.

Deferred config additions for later phases:

- `healthcheck-url`
- `deploy.env-file` with default `deploy/default.env`
- `hooks.pre-start`
- `hooks.post-start`

## Deploy Repository Contract

The deploy workflow expects these repository artifacts:

- `Dockerfile`
- `compose.yaml`
- `compose.server.yaml`

Deferred repository artifacts for later phases:

- `deploy/default.env` by default, unless overridden in config
- `lexe-logs.md`

## Non-Goals In Scope For This Spec

- No blue/green or canary deploy behavior
- No automatic rollback on deploy failure
- No dry-run mode
- No custom-domain concerns

## Planned Validation

- Re-running deploy for the same commit to confirm the no-op path once that behavior exists
- Forced redeploy of the same release once that behavior exists
- Re-running a dirty deploy for the same commit to confirm it rebuilds and re-transfers
- Pre-start hook failure handling
- HTTP healthcheck failure handling beyond Compose health waiting
- Successful deploy log entry creation

# SSH adjustments

## Summary

The current SSH handling for exe.dev usage, especially in integration tests, is too messy and too
split across different code paths. We should move to explicit CLI options and a single source of
truth for SSH identity and host-key behavior.

## Goals

- Replace environment-variable-driven integration test behavior with explicit CLI options.
- Centralize SSH identity and host-key behavior so provision, deploy, status, and related helpers
  all behave consistently.
- Keep normal CLI usage secure by default.
- Keep integration tests simple and avoid churn in the user's `known_hosts` file.

## CLI options

Add or complete these CLI options:

- `-i / --ssh-key`: `click.Path` using `pathlib.Path`; if given, it must exist; default `None`
- `--ssh-host-key-check/--no-ssh-host-key-check`: default `True`
- `--ssh-known-hosts-manage`: flag; default `True`

## Option semantics

- If `ssh_host_key_check` is `True`, `ssh_known_hosts_manage` controls whether we maintain entries
  in the user's `known_hosts` file for managed exe.dev VMs.
- If `ssh_host_key_check` is `False`, `ssh_known_hosts_manage` must also be `False`.
- The combination `--no-ssh-host-key-check --ssh-known-hosts-manage` is invalid and should raise a
  user-facing CLI error.

## Config plumbing

- Keep CLI-derived SSH options in `CLIOpts`.
- Pass config file data and CLI options together via `ConfigOpts(config=..., opts=...)`.
- `Provision`, `Deploy`, `Status`, and related helpers should accept that combined config object
  rather than relying on environment variables or ad hoc parameter passing.

## Single source of truth for SSH behavior

There should be one shared builder for SSH identity and host-key behavior. It must be used for all
exe.dev SSH-related paths, including direct SSH calls and Docker-based remote operations.

The shared behavior must cover:

- which identity file to use
- whether host key checking is enabled
- whether `known_hosts` is managed
- any SSH options needed by plain `ssh`
- any SSH options or environment needed by Docker remote operations

This shared logic must not depend on the destination hostname appearing in a specific argument
position.

## Normal CLI behavior

For normal CLI usage:

- `ssh_host_key_check` defaults to `True`
- `ssh_known_hosts_manage` defaults to `True`

When `ssh_known_hosts_manage` is `True`:

- after successful provision, fetch the VM's `/etc/ssh/ssh_host_ed25519_key.pub`
- construct a `known_hosts` entry using the VM hostname only
- append that entry to the user's `known_hosts`
- on destroy, remove the VM hostname entry or entries from the user's `known_hosts` using
  `ssh-keygen -R ...`

Lifecycle decisions:

- do not remove old `known_hosts` entries before adding a new one
- cleanup happens on destroy only
- after failed provision, do nothing
- hostname only; do not include IP address

VM name reuse is acceptable because the old hostname entry should have been removed during destroy.

## Integration test behavior

Integration tests should set `CLIOpts` directly instead of manipulating environment variables.

For integration tests:

- `ssh_host_key_check = False`
- `ssh_known_hosts_manage = False`
- use the dedicated test SSH key from `tests/ssh-keys/`

The purpose is to avoid churn in the user's `known_hosts` file for short-lived test VMs.

## Validation expectations

Update or add tests to cover:

- CLI option parsing and invalid option combinations
- propagation of `CLIOpts` through `ConfigOpts`
- shared SSH behavior in provision, deploy, and status flows
- integration-test behavior without environment-variable toggles
- known-hosts add behavior after successful provision
- known-hosts removal behavior on destroy
